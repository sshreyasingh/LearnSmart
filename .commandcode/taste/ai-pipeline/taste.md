# ai-pipeline
- AI is only permitted for: project purpose analysis, database explanation, authentication explanation, and AI chatbot. Everything else (code metrics, security scanning, performance analysis, dependency graphs, tech stack detection, folder structure, API detection, executive summary, recommendations, architecture review, learning roadmap, gap analysis) must use deterministic parsers. Even AI features should only run on explicit user request — never automatically. Confidence: 0.80
- Use OpenRouter API with DeepSeek models (not Gemini or NVIDIA) for all AI analysis. Confidence: 0.80
- The code analysis pipeline must be RAG-based: clone repository → chunk source files intelligently by functions/classes/modules → generate embeddings → store in ChromaDB with metadata (file path, language, function name, class name, dependencies, project ID) → use semantic search to retrieve relevant chunks for AI context. Confidence: 0.80
- Use hierarchical summarization: function summary → file summary → folder summary → project summary. Feed summaries (not raw code) into the LLM to reduce token usage by 80-90%. Confidence: 0.75
- Cache AI responses and store embeddings in ChromaDB for reuse to reduce token usage. Confidence: 0.70
- Tier AI models by task complexity: metadata extraction (no AI) → embeddings (small model) → summarization (inexpensive model) → final explanation (best model). Don't use the best model for every task. Confidence: 0.85
- Code metrics must deterministically analyze: routes, components, comments, and cyclomatic complexity. Remove any code-metric analyses that are not being performed. Confidence: 0.75
- Include an AI chat section on the analysis page where users can ask doubts and questions about their project. During AI chat, use RAG to retrieve only the most relevant code chunks (not the entire project) and send them to DeepSeek via OpenRouter to minimize token usage. Confidence: 0.70
- Keep the dependency graph / architecture graph visible on the analysis page — do not remove it when simplifying the analysis UI. Confidence: 0.70
- Interview questions must be generated using static code analysis and predefined question banks based on detected technologies, frameworks, auth, database, APIs, architecture, and project structure — never use AI/LLMs for question generation. Confidence: 0.85
- Interview questions should be auto-generated after project analysis completes, and the interview section should appear below the learning resources section on the analysis page. Confidence: 0.75
- Use LangChain.js to orchestrate the RAG pipeline: connect ChromaDB retrieval with OpenRouter (DeepSeek), manage prompt templates, retrieval chains, conversation memory, and context assembly. Use LangChain only for AI orchestration, not for static analysis. Confidence: 0.70
- When using AI to understand a project's purpose and features, analyze only the important/relevant files (package.json, README, main entry points, config files, key source directories) — do not feed the entire project to the AI. Focus on files that reveal what the project does and why. Confidence: 0.65
